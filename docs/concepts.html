Concepts & Background

This page explains every technology used in the project from scratch. No prior knowledge assumed.

What is SPIFFE?

SPIFFE (Secure Production Identity Framework For Everyone) is an open standard that defines how software services identify themselves to each other. Think of it as an "identity card" system for applications.

Instead of hard-coding passwords or API keys, SPIFFE gives each workload a cryptographic identity called a SPIFFE ID — a URI like:

spiffe://example.org/ns/mcp-demo/sa/mcp-client
Why it matters: Traditional client_secret-based authentication requires storing, rotating, and distributing secrets. SPIFFE eliminates this by tying identity to the workload's runtime environment — if the pod is running with the correct service account on an attested node, it automatically gets its identity.

SPIFFE ID Format

PartExampleMeaning
spiffe://schemeAlways starts with this
example.orgtrust domainThe root of trust — who controls identity issuance
/ns/mcp-demopath segmentKubernetes namespace
/sa/mcp-clientpath segmentKubernetes service account

What is SPIRE?

SPIRE (SPIFFE Runtime Environment) is the reference implementation of SPIFFE. It consists of two main components:

SPIRE Server

The central authority. Maintains a registry of which workloads should get which identities. Signs SVIDs using its Certificate Authority (CA). Runs as a StatefulSet in Kubernetes.

SPIRE Agent

Runs on every node (DaemonSet). Performs workload attestation — verifies that a pod is who it claims to be by checking its Kubernetes metadata (namespace, service account, labels). Delivers SVIDs to pods via the Workload API socket.

How SPIRE Issues Identities

  1. Pod starts with a volume mount for the Workload API socket (via SPIFFE CSI Driver)
  2. Application calls the Workload API: "Give me a JWT SVID for audience X"
  3. SPIRE Agent verifies the calling process matches a registered entry (namespace + service account + labels)
  4. Agent requests a signed SVID from SPIRE Server
  5. Server signs a JWT with the workload's SPIFFE ID and returns it
  6. Application receives a short-lived JWT (5 min TTL) to use as a credential

What is an SVID?

An SVID (SPIFFE Verifiable Identity Document) is the cryptographic proof of a workload's identity. There are two types:

TypeFormatUsed In This Project
JWT SVIDJSON Web Token (signed JWT)Yes — used as client_assertion for OAuth
X.509 SVIDX.509 certificate with SPIFFE ID in SANNo (not needed for this demo)

Anatomy of a JWT SVID

{
  "header": {
    "alg": "RS256",
    "kid": "abc123...",
    "typ": "JWT"
  },
  "payload": {
    "sub": "spiffe://example.org/ns/mcp-demo/sa/mcp-client",
    "aud": ["http://keycloak.mcp-demo.svc.cluster.local:8080/realms/mcp-demo"],
    "iss": "https://oidc-discovery.example.org",
    "exp": 1710600300,
    "iat": 1710600000
  }
}

What is Keycloak?

Keycloak is an open-source Identity and Access Management (IAM) server. In this project, it serves as the OAuth2 Authorization Server — the gatekeeper that decides whether a client should receive an access token.

Keycloak Concepts Used

ConceptDescription
RealmAn isolated authentication domain. Ours is called mcp-demo.
ClientAn application registered in the realm. Our client ID is the SPIFFE ID itself.
SPI (Service Provider Interface)Keycloak's plugin system. We add two custom SPIs for SPIFFE support.
Authentication FlowA sequence of steps Keycloak follows to verify a client. We add a "spiffe-clients" flow.
Client AuthenticatorThe mechanism a client uses to prove its identity. Normally client_secret; we replace it with client-spiffe-jwt.

Custom SPIs

SVID Client Authenticator SPI

Intercepts token requests where client_assertion_type is urn:ietf:params:oauth:client-assertion-type:jwt-spiffe. Fetches JWKS from SPIRE OIDC Discovery, validates the JWT SVID signature, checks issuer/audience/expiry, and maps the SPIFFE ID to a registered client.

SPIFFE DCR SPI

Provides a Dynamic Client Registration endpoint at /clients-registrations/spiffe-dcr/register. Accepts a JWT SVID as a "software statement", verifies it, and auto-registers an OAuth client with the SPIFFE ID as the client ID.

What is MCP?

MCP (Model Context Protocol) is an open protocol that defines how AI agents (like Claude) interact with external tools and data sources. Key authorization requirements from the MCP spec:

  • MCP clients SHOULD use Dynamic Client Registration (RFC 7591)
  • MCP servers MUST implement Protected Resource Metadata (RFC 9728)
  • Clients MUST implement Resource Indicators (RFC 8707)
Our innovation: By using SPIFFE, we replace anonymous DCR with cryptographically-verified identity-based registration. The MCP client doesn't just register — it proves who it is.

OAuth 2.0 & OIDC Basics

OAuth 2.0 is the authorization framework. OpenID Connect (OIDC) is an identity layer built on top. Here's how they apply:

ConceptRole in This System
Authorization ServerKeycloak — issues access tokens
Client Credentials GrantThe OAuth flow used — machine-to-machine, no user involved
client_assertionThe JWT SVID replaces client_secret
client_assertion_typeurn:ietf:params:oauth:client-assertion-type:jwt-spiffe
Access TokenThe result — a Bearer JWT that authorizes API calls
JWKS (JSON Web Key Set)Public keys published at a URL, used to verify JWT signatures

OIDC Discovery & JWKS

The SPIRE OIDC Discovery Provider is a bridge between SPIRE and standard OIDC-aware systems (like Keycloak):

  • Serves /.well-known/openid-configuration — tells clients where to find keys
  • Serves /keys (JWKS endpoint) — contains the public keys SPIRE uses to sign SVIDs
  • Keycloak's SPIFFE SPI fetches from this JWKS to verify incoming JWT SVIDs

Workload API & CSI Driver

The Workload API is a Unix domain socket that SPIRE Agent exposes to pods. The SPIFFE CSI Driver automatically mounts this socket into pods that request it:

volumes:
  - name: spiffe-workload-api
    csi:
      driver: "csi.spiffe.io"
      readOnly: true

Inside the pod, the socket is available at /spiffe-workload-api/spire-agent.sock.
