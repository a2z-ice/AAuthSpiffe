Troubleshooting — 12 Issues Solved

Every implementation issue encountered during deployment, documented with symptom, root cause, fix, and affected files.

General Debugging Approach

Tiered diagnosis — do not blindly retry failed commands. 
# Tier 1 (5s): Check pod status
kubectl get pods -n <namespace> | grep -v Running

# Tier 2 (10s): Check events
kubectl get events -n <namespace> --sort-by=.lastTimestamp | tail -10

# Tier 3 (15s): Check logs
kubectl logs <pod> --tail=30 -n <namespace>

# Tier 4 (30s): Full resource details
kubectl describe <resource> -n <namespace> | tail -40

Implementation Issues

1 ImagePullBackOff on keycloak-spiffe
Symptom
Keycloak pod stuck in ImagePullBackOff or ErrImagePull state after applying keycloak.yaml.
Root Cause
The custom keycloak-spiffe:latest image was built locally but not loaded into the Kind cluster. Kind nodes cannot pull from the local Docker daemon directly.
Fix
kind load docker-image keycloak-spiffe:latest --name spiffe-mcp-demo
kubectl -n mcp-demo delete pod -l app=keycloak  # Force re-pull
Affected Files
manifests/keycloak.yaml — uses imagePullPolicy: Never to prevent pulling from registry
2 SPIRE Agent CrashLoopBackOff
Symptom
SPIRE Agent pods in CrashLoopBackOff immediately after Helm install.
Root Cause
SPIRE Agent requires a healthy SPIRE Server to complete node attestation. During initial deployment, the Server takes 30-60s to become ready, and the Agent crashes during that window.
Fix
Wait 60 seconds. The Agent will auto-recover once the Server is healthy. Verify with: 
kubectl -n spire-system get pods -w  # Watch until all Running
Affected Files
infra/spire-values.yaml — Agent liveness probes handle restart
3 CSI Driver FailedMount
Symptom
MCP Client pod stuck in ContainerCreating with FailedMount event for spiffe-workload-api volume.
Root Cause
The SPIFFE CSI Driver or SPIRE Agent DaemonSet was not running on the node where the pod was scheduled.
Fix
# Check CSI driver and agent are on all nodes
kubectl -n spire-system get ds
# Verify the worker node has the spire-agent label
kubectl get nodes --show-labels | grep spire-agent
Affected Files
infra/kind-config.yaml — worker node needs spire-agent: "true" label
4 Keycloak Connection Refused to PostgreSQL
Symptom
Keycloak logs show Connection refused to postgres.mcp-demo.svc.cluster.local:5432.
Root Cause
Keycloak started before PostgreSQL was ready. The StatefulSet has no dependency ordering.
Fix
Always deploy and wait for PostgreSQL first: 
kubectl apply -f manifests/postgres.yaml
kubectl -n mcp-demo wait --for=condition=ready pod -l app=postgres --timeout=120s
# Then deploy Keycloak
Affected Files
deploy-all.sh — enforces ordering
5 OIDC Discovery Returns 404
Symptom
curl http://localhost:30443/.well-known/openid-configuration returns 404 or connection refused.
Root Cause
The OIDC Discovery Provider pod was not running, or the NodePort service was not applied.
Fix
# Check OIDC provider pod
kubectl -n spire-system get pods | grep oidc
# Apply NodePort if missing
kubectl apply -f manifests/oidc-nodeport.yaml
Affected Files
manifests/oidc-nodeport.yaml, infra/spire-values.yaml (domains config)
6 client-spiffe-jwt Provider Not Found in Keycloak
Symptom
Running configure-keycloak.sh fails with "Execution not found" when adding the SPIFFE authenticator.
Root Cause
The custom SPI JARs were not properly copied into /opt/keycloak/providers/ during the Docker build, or kc.sh build was not run.
Fix
Rebuild the Keycloak image ensuring the COPY and RUN build steps succeed: 
docker build --no-cache -f docker/Dockerfile.keycloak-custom -t keycloak-spiffe:latest .
kind load docker-image keycloak-spiffe:latest --name spiffe-mcp-demo
kubectl -n mcp-demo rollout restart statefulset/keycloak
Affected Files
docker/Dockerfile.keycloak-custom
7 Token Exchange Returns 400: Invalid client_assertion_type
Symptom
Token request returns 400 Bad Request with "invalid client_assertion_type" error.
Root Cause
The client_assertion_type value must exactly match what the SPI expects. The spec references both jwt-spiffe and jwt-bearer variants.
Fix
The SPI uses urn:ietf:params:oauth:client-assertion-type:jwt-bearer as the assertion type. The realm's authentication flow must include the client-spiffe-jwt execution as ALTERNATIVE: 
./scripts/configure-keycloak.sh  # Sets up the flow correctly
Affected Files
scripts/configure-keycloak.sh, scripts/test-spiffe-auth.sh
8 Single-Use Token Rejection (Replay Detection)
Symptom
First token request succeeds, but subsequent requests with the same JWT SVID return 400 or 401.
Root Cause
The SPIFFE SPI enforces single-use tokens using issuer+subject+iat as a replay detection key. If two requests share the same iat (issued-at timestamp, second precision), the second is rejected.
Fix
Fetch a fresh JWT SVID for each token request. Add a 1-second delay between requests to ensure different iat values: 
import time, uuid
time.sleep(1)  # Ensure unique iat
svid = client.fetch_jwt_svid(
    audience={audience, f"nonce-{uuid.uuid4().hex[:8]}"}
)
Affected Files
tests/test_05_auth_flows.py, tests/test_06_e2e_flows.py, tests/test_07_chaos.py
9 Keycloak JWKS URL Misconfiguration
Symptom
Token exchange fails with a signature verification error in Keycloak logs.
Root Cause
The SPIFFE client in Keycloak had the wrong JWKS URL configured. It needs to point to the OIDC Discovery Provider's HTTP proxy (not the HTTPS endpoint directly), because the internal service may have self-signed TLS.
Fix
The configure-keycloak.sh script sets the correct URL: 
JWKS_URL="http://oidc-http-proxy.spire-system.svc.cluster.local/keys"
Set on both the realm attributes (spiffe.jwks.url) and the client attributes (jwks.url, use.jwks.url=true).
Affected Files
scripts/configure-keycloak.sh, manifests/realm-config.json
10 Helm Install Timeout on PVC Provisioning
Symptom
helm upgrade --install spire hangs and eventually times out.
Root Cause
SPIRE Server needs a PVC, and Kind needs a default StorageClass. If none exists, PVC stays Pending forever.
Fix
# Verify default StorageClass exists
kubectl get sc
# Kind usually creates "standard" as default. If missing:
kubectl annotate sc standard storageclass.kubernetes.io/is-default-class=true
Affected Files
infra/spire-values.yaml — persistence section
11 Keycloak Health Endpoint on Port 9000
Symptom
curl http://localhost:30080/health/ready returns 404 even though Keycloak is running.
Root Cause
Keycloak 26.x moved the management/health endpoints to port 9000 by default. The NodePort service only exposes port 8080.
Fix
Use the realm endpoint to verify health instead: 
# This works on port 8080:
curl -s http://localhost:30080/realms/master | jq .realm
# Returns "master" if healthy
The readinessProbe in keycloak.yaml uses /health/ready on port 8080, which works because Keycloak still serves this path on the main port in dev mode.
Affected Files
manifests/keycloak.yaml (readinessProbe), test files
12 Audience Mismatch Between SVID and Keycloak
Symptom
Token exchange returns 401 with "audience mismatch" even with a valid SVID.
Root Cause
The JWT SVID's aud claim must match the Keycloak realm's issuer URL. If you request the SVID with audience=http://localhost:30080/realms/mcp-demo (host URL) but the SPI checks against the internal URL, they won't match.
Fix
Fetch the audience dynamically from Keycloak's OIDC discovery: 
resp = requests.get(f"{KC_URL}/realms/{REALM}/.well-known/openid-configuration")
audience = resp.json()["issuer"]  # Use this as the SVID audience
Affected Files
tests/test_04_jwt_svid.py, tests/test_05_auth_flows.py
