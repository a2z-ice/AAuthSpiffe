Manual Positive Testing Guide

5 scenarios to verify the system works correctly.

Scenario 1: Fetch JWT SVID

kubectl -n mcp-demo exec deploy/mcp-client -- python3 -c "
from spiffe import WorkloadApiClient
c = WorkloadApiClient('unix:///spiffe-workload-api/spire-agent.sock')
s = c.fetch_jwt_svid(audience={'http://keycloak.mcp-demo.svc.cluster.local:8080/realms/mcp-demo'})
print('SPIFFE ID:', s.spiffe_id)
print('Token:', s.token[:80], '...')
print('Expiry:', s.expiry)
"

Expected: SPIFFE ID = spiffe://example.org/ns/mcp-demo/sa/mcp-client, token is a 3-part JWT, expiry ~5 min ahead.

Scenario 2: Token Exchange

kubectl -n mcp-demo exec deploy/mcp-client -- bash /scripts/test-spiffe-auth.sh

Expected: "SUCCESS! Got access token:" with token_type=Bearer.

Scenario 3: Verify OIDC Discovery

# From host machine
curl -sk https://localhost:30443/.well-known/openid-configuration | jq .
curl -sk https://localhost:30443/keys | jq '.keys | length'

Expected: OIDC config with issuer and jwks_uri. At least 1 key in JWKS.

Scenario 4: Keycloak Admin Console

# Open in browser
open http://localhost:30080/admin
# Login: admin / admin
# Navigate: Clients > spiffe://example.org/ns/mcp-demo/sa/mcp-client

Expected: Client exists with authenticator type client-spiffe-jwt.

Scenario 5: Full Test Suite

./scripts/run-all-tests.sh

Expected: 80 tests passed, 0 failed.

Manual Negative Testing Guide

6 scenarios to verify the system rejects invalid requests.

Scenario 1: Garbage Token

curl -s -X POST http://localhost:30080/realms/mcp-demo/protocol/openid-connect/token \
  -d "grant_type=client_credentials" \
  -d "client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer" \
  -d "client_assertion=not.a.real-jwt" | jq .

Expected: 400 or 401 error response.

Scenario 2: Empty Assertion

curl -s -X POST http://localhost:30080/realms/mcp-demo/protocol/openid-connect/token \
  -d "grant_type=client_credentials" \
  -d "client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer" \
  -d "client_assertion=" | jq .

Expected: 400 or 401 error response.

Scenario 3: Wrong Assertion Type

curl -s -X POST http://localhost:30080/realms/mcp-demo/protocol/openid-connect/token \
  -d "grant_type=client_credentials" \
  -d "client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-spiffe" \
  -d "client_assertion=eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiJ0ZXN0In0.fake" | jq .

Expected: 400 — unrecognized assertion type (SPI uses jwt-bearer, not jwt-spiffe).

Scenario 4: Missing grant_type

curl -s -X POST http://localhost:30080/realms/mcp-demo/protocol/openid-connect/token \
  -d "client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer" \
  -d "client_assertion=some.jwt.token" | jq .

Expected: 400 — missing grant_type parameter.

Scenario 5: Invalid DCR Software Statement

curl -s -X POST http://localhost:30080/realms/mcp-demo/clients-registrations/spiffe-dcr/register \
  -H "Content-Type: application/json" \
  -d '{"software_statement": "invalid.jwt.token"}' | jq .

Expected: 400 or 401 — invalid software statement.

Scenario 6: JWKS Private Key Check

curl -sk https://localhost:30443/keys | jq '.keys[] | keys' | grep -E '"d"|"p"|"q"'

Expected: No output — private fields must never appear in JWKS.
