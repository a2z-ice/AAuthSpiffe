Product Manual

A non-technical guide to understanding, deploying, and using the SPIFFE + MCP OAuth authentication system.

What This System Does

Imagine you have AI agents (like chatbots or automated tools) that need to access protected services. Normally, you'd give each agent a password (called a "client secret") to prove who they are. But passwords can be leaked, stolen, or forgotten to rotate.

This system replaces passwords with cryptographic identity cards. Each AI agent automatically receives a short-lived digital ID card (valid for just 5 minutes) that proves who they are based on where they're running — not based on a stored secret.

The Old Way

  • Store a password in each agent's config
  • Rotate passwords manually every few months
  • Risk: leaked passwords grant full access
  • No way to know which specific agent used a password

The New Way (This System)

  • No passwords stored anywhere
  • Identity cards auto-issued and auto-rotated every 5 minutes
  • Even if stolen, useless after 5 minutes
  • Every request tied to a specific agent identity

Deploying the System

One command does everything. Open Terminal, navigate to the project folder, and run: 
./deploy-all.sh

This takes approximately 10-15 minutes and will:

  1. Create a local Kubernetes cluster
  2. Install the identity system (SPIRE)
  3. Build custom security components
  4. Deploy the authentication server (Keycloak)
  5. Deploy the AI agent demo workloads

You'll see progress messages for each step. Wait for the final message:

============================================
  Deployment Complete!
============================================

Verifying It Works

After deployment, verify the system is working with these simple checks:

Check 1: Is everything running?

# This should show pods in "Running" state
kubectl -n mcp-demo get pods
kubectl -n spire-system get pods

All pods should show Running in the STATUS column.

Check 2: Run the authentication demo

kubectl -n mcp-demo exec -it deploy/mcp-client -- bash /scripts/test-spiffe-auth.sh

You should see:

  • "Step 2: Fetch JWT SVID" — shows the agent got its identity card
  • "Step 3: Authenticate to Keycloak" — shows "SUCCESS! Got access token"

Check 3: Run the full test suite

./scripts/run-all-tests.sh

Expected result: "80 passed, 0 failed"

Shutting Down

When you're done, clean everything up with one command:

./teardown.sh

This completely removes the cluster and all custom images. Nothing is left on your machine except the project files.

Frequently Asked Questions

Q: Does this install anything permanently on my Mac?

No. Everything runs inside Docker containers. The teardown.sh script removes all containers and images. The only things on your Mac are Docker Desktop and the command-line tools (kind, kubectl, etc.).

Q: How much disk space does this use?

Approximately 5-8 GB of Docker images and container data. This is freed when you run teardown.sh.

Q: Can I run this on Windows or Linux?

The system is designed for macOS but the underlying technology (Docker + Kind) works on Windows and Linux too. Minor path adjustments may be needed.

Q: Is this production-ready?

No, this is a demo/development setup. For production you'd need: TLS everywhere, proper certificate management, high-availability configurations, production-grade PostgreSQL, and stricter RBAC. The architecture and authentication pattern are production-ready; the deployment configuration is simplified for demo purposes.

Q: What happens if the 5-minute identity card expires?

The agent simply requests a new one. This happens automatically. There's no manual intervention needed — it's like getting a new boarding pass every 5 minutes.

Q: What if Docker Desktop runs out of memory?

Open Docker Desktop > Settings > Resources, and increase memory to at least 8 GB. The system needs this for Keycloak + SPIRE + PostgreSQL to run comfortably.

Q: How do I see the authentication happening in real-time?

Run the demo script and watch the output:

kubectl -n mcp-demo exec -it deploy/mcp-client -- bash /scripts/test-spiffe-auth.sh

It shows each step: fetching the identity card, sending it to Keycloak, and receiving the access token.

Q: Who made this?

This implementation follows the reference architecture by Christian Posta and the IETF draft draft-ietf-oauth-spiffe-client-auth. The deployment automation and test suite were built specifically for this project.
