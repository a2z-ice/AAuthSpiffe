Authentication Flows Explained

Two authentication flows: Flow B (Token Exchange) for obtaining access tokens, and Flow A (Dynamic Client Registration) for automatic client setup.

Flow B: Token Exchange (Primary Flow)

The MCP client uses a JWT SVID as a client_assertion to obtain an OAuth access token from Keycloak — no client_secret needed.

Step-by-Step with Code

Step 1-2: Fetch JWT SVID from SPIRE

from spiffe import WorkloadApiClient

# Connect to SPIRE Agent via Workload API (mounted by CSI Driver)
client = WorkloadApiClient("unix:///spiffe-workload-api/spire-agent.sock")

# Request JWT SVID with audience = Keycloak realm URL
keycloak_url = "http://keycloak.mcp-demo.svc.cluster.local:8080"
realm = "mcp-demo"
audience = f"{keycloak_url}/realms/{realm}"

jwt_svid = client.fetch_jwt_svid(audience={audience})
print(f"SPIFFE ID: {jwt_svid.spiffe_id}")
print(f"Token (first 80 chars): {jwt_svid.token[:80]}...")
print(f"Expires: {jwt_svid.expiry}")  # ~5 minutes from now

Step 3-6: Exchange SVID for Access Token

import requests

token_url = f"{keycloak_url}/realms/{realm}/protocol/openid-connect/token"

# POST to token endpoint: JWT SVID replaces client_secret
response = requests.post(token_url, data={
    "grant_type": "client_credentials",
    "client_assertion_type": "urn:ietf:params:oauth:client-assertion-type:jwt-spiffe",
    "client_assertion": jwt_svid.token,
    "scope": "openid"
})

if response.status_code == 200:
    data = response.json()
    print(f"access_token: {data['access_token'][:60]}...")
    print(f"token_type: {data['token_type']}")      # Bearer
    print(f"expires_in: {data['expires_in']}s")
else:
    print(f"FAILED: {response.status_code} {response.text}")

What Keycloak's SPIFFE SPI Does (Steps 4-5)

  1. Extracts the client_assertion from the POST body
  2. Decodes the JWT header to get the kid (Key ID)
  3. Fetches JWKS from the SPIRE OIDC Discovery Provider at the configured URL
  4. Matches the JWT's kid to a public key in the JWKS
  5. Verifies the JWT signature using the matched public key
  6. Validates claims: iss matches trust domain, sub is a valid SPIFFE ID, aud matches realm URL, exp is in the future
  7. Enforces single-use: rejects JWTs with the same iss+sub+iat combination (replay protection)
  8. Maps the sub (SPIFFE ID) to a registered Keycloak client
  9. Issues an access token for the matched client

Flow A: Dynamic Client Registration (DCR)

Automatic OAuth client creation using a SPIFFE software statement. No manual client setup in Keycloak admin needed.

DCR Code Example

import requests

keycloak_url = "http://keycloak.mcp-demo.svc.cluster.local:8080"
realm = "mcp-demo"
dcr_url = f"{keycloak_url}/realms/{realm}/clients-registrations/spiffe-dcr/register"

# JWT SVID enriched with software statement claims by SPIRE's
# CredentialComposer plugin (adds jwks_url, client_auth claims)
response = requests.post(dcr_url, json={
    "software_statement": jwt_svid.token
}, headers={"Content-Type": "application/json"})

if response.status_code in [200, 201]:
    data = response.json()
    print(f"client_id: {data.get('clientId')}")
    print(f"auth_method: {data.get('clientAuthenticatorType')}")

Critical security property: SPIRE's signing private key never leaves the SPIRE Server. Keycloak only accesses the public key via JWKS. This is the same trust model used by Google, Auth0, and every OIDC provider.

IETF Draft Reference

This implementation follows draft-ietf-oauth-spiffe-client-auth-01 (March 2026), which defines:

  • A new client_assertion_type value: urn:ietf:params:oauth:client-assertion-type:jwt-spiffe
  • Validation rules for SPIFFE JWT SVIDs used as client assertions
  • Integration with OAuth 2.0 Dynamic Client Registration
  • JWKS-based trust establishment between SPIRE and OAuth Authorization Servers
